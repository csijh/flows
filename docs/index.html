Flows: A framework for logical web servers

There are various frameworks for creating JavaScript web servers, the best known being Express. There are two potential problems with such servers. First, they are based on callbacks, which can make them difficult to understanding or test. Second, the design of their components is often obscure, with complex hidden code and many options. This may be suitable for non-programmers or programmers who don't want to be bothered with how things work, but is less helpful for serious learners.

A flows web server is built out of components. The aim is to make it easy to understand, adapt, write, combine and test components. Here's a quick summary of the conventions for components:

  • a component is a function, usually an async function
  • it takes request and response objects as arguments
  • it returns nothing (except a confirmation of completion if async)
  • it makes changes to the request and response objects
  • errors cause response.status to be set
  • most components do nothing if response.status is already set

The main differences from express 'middleware' components, for example, are:

  • there are no callback arguments, making components self-contained and cleanly composable
  • components are small, simple and easy to understood or adapt
  • the entire state of processing is visible on the request and response objects so that it easy to track
  • the error conventions almost eliminate the need for explicit error handling.

Normalizing

The flows.mjs library contains a normalize function which validates and normalizes the URL in the request. Here's a norm.mjs program to try it out without creating a server. It is assumed that flows.mjs is in the same directory:

import flows from "./flows.mjs";

let request = { url: "//products/././pets/../dogs.html?x=1&y=2" };
let response = {};

let go = flows.normalize("index.html");
go(request, response);

console.log("request =", request);
console.log("response =", response);

The program uses standard JavaScript modules rather than node's own module conventions, and currently must be run with the --experimental-modules option:

> node --experimental-modules norm.mjs
request = { url: '/products/dogs.html', parameters: { x: '1', 'y': '2' } }
reponse = {}

The normalize fucntion takes one argument, which is the name of the default index file in a directory, top be added to a URL ending with the / character. It returns a component, in this case called go. The go component is an ordinary function. It is demonstrated by passing it a request object with a url field which the component updates. The parameters are taken off the URL, pulled apart and put into a request.parameters object, and then the URL is tidied up so that it becomes suitable for use as a file path, for example. In the process, some basic URL validation is done to guard against security attacks:

let request = { url: "/products/../../secret.txt" };
...
> node --experimental-modules norm.mjs
request = { url: '/../secret.txt' }
response = { status: 400,
  message: 'Url refers to a file outside of the site' }

The validation error is recorded in the response object, which will cause most other components to be bypassed, until the error is finally delivered back to the client. The normalize component does not currently do any translation of encodings such as %20, so only simple plain text file names without spaces are supported.

In general, components can be tested independently of running a web server, by looking at their effect on the request and response objects at each step.

Reading and writing files

The read function from flows.mjs can be illustrated with this read.mjs program:

import { read } from "./flows.mjs";

let fn = read("a.txt");

async function go(request, response) {
    await fn(request, response);
    console.log(response);
}

go({}, {});

This illustrates that items can be imported from flows.mjs individually, so they can be used without a prefix. The read function takes one argument, a file path, and returns a component for reading the file. The component, called fn above, has to be called using await, because it involves asynchronous I/O, otherwise the response would be printed out before the file was read in. And the call has to be wrapped in a function which is marked as async, because it contains await. The content of the file is stored in response.data, or appended to it if it already exists.

The read function accepts file paths containing parameters which refer to fields of the request object. For example, read("/.files{url}") indicates the request.url should be appended to ./files to form the full file path. This allows read to use file paths which are partly or wholly calculated during previous components.

There is a write component which takes a file path and writes response.data to the given file. It too accepts parameters representing request fioelds.

Combining components

The example program which reads a file can be simplified like this:

import { chain, read, log } from "./flows.mjs";

let go = chain(read("a.txt"), log);

go({}, {});

Components can be combined to make more complex components. The component go is built by combining two other components in a chain. The chain function combines any number of components in sequence. The log component makes testing more convenient by including logging in the component sequence, avoding the need for explicit use of await.

When go is called, the effect is to call the components in the chain one after another, passing the request and response to each, and waiting for each to finish before calling the next. The go function above doesn't need to be called with await, because there is nothing following it which needs to wait for it to finish.

Error handling

Here's an example program cat.mjs which reads in two files, concatenates them, and writes the result to a third file:

import { read, write, chain, log } from "./flows.mjs";

let go = chain(read("a.txt"), read("b.txt"), write("c.txt"), log);

go({}, {});

If the first file a.txt doesn't exist, the second read and the write are bypassed, and the error is reported at the end. Most ordinary components pass on an error rather than taking any action, so that a single pipeline of components can handle all cases.

Making choices

The choose function is used to make decisions:

import { choose, log } from "./flows.mjs";

let go = choose("/admin/*", log);

go({ url:"/admin/index.html" }, {});

If choose is called with two arguments, it acts like an if...then... statement. If the URL in the request matches the pattern, then the go component calls the log component, otherwise the go component does nothing.

If choose is called with three arguments, it acts like an if...then...else... statement. For example, this component will execute either c1 or c2.

let go = choose("/admin/*", c1, c2);

The choose function can be called with any number of pattern/component pairs and an optional default component. It acts like a switch statement, calling the first component whose pattern matches:

let go = choose("/admin/*", login, "/do/*", act, display);

A URL pattern must contain a single asterisk, representing arbitrary text. It can thus pick out specific prefixes, or specific suffixes, or both. The pattern can also be preceded by a request method, e.g. "GET /admin/*", to match only requests with that method. The method name can be upper or lower case.

Content type

The type function deals with finding the right content type for the response. It takes one argument which is a map object with a getType method. It returns a component that sets the response.type field. For example:

import { normalize, log, chain, type } from "./flows.mjs";

let map = { html: "text/html" };
map.getType = function(ext) { return this[ext]; }

let go = chain(normalize, type(map), log);

go({ url: "/products/dogs.html?x=1&y=2" }, {});

A map object with a single entry is used to illustrate the function. In a real web server, an extensive map is usually needed, e.g. by installing the mime module with npm install mime and importing it with:

import map from "mime/lite";

The type component does nothing if the response.type field is already set. If a status other than 200 (OK) has been set, the error message is used as the response data and the type is set to "text/plain". If the type for .html is "application/xhtml+xml" then content negotiation is used to check that the type is accepted by the client, otherwise the type is changed to "text/html".

Delivery

The deliver component sends the response back to the client. To demonstrate it without building a server, a response object has to be provided with writeHeader, write and end methods:

import { deliver } from "./flows.mjs";

let request = {};
let response = {
    status: 200, data: "xyz", type: "text/plain",
    writeHead, write, end
};
function writeHead(status, head) { console.log("writing head", status, head); }
function write(data) { console.log("writing data", data); }
function end() { console.log("sending response"); }

deliver(request, response);

A complete server

A complete static server can be defined using the components described so far:

import http from "http";
import map from "mime/lite";
import { chain, normalize, read, type, deliver, log } from "./flows.mjs";

let handle = chain(
    normalize("index.html"),
    read("./files{url}"),
    type(map),
    deliver
);

let service = http.createServer(handle);
service.listen(8080, "localhost");

Future components

These could include components to:

  • read the body of a post request
  • interpret the body of a post request as parameters
  • read the body of a put request, and store it as a file
  • tell the browser to redirect
  • use cookies to manage sessions, redirecting to a login page if necessary
  • check authorisation.
  • rewrite the url so that the first part becmoes a template page and later parts become parameters
  • arrange to call a specific handler
  • treat one part of a url as a handler name, and pass control to that handler
  • set a header on the response
  • read data from a database
  • update a database
